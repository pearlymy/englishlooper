# Browser WAV Seeking Bug & Hybrid Playback Architecture

When playing and seeking WAV files (especially mono PCM WAV files generated by audio compression or uploaded by the user) via blob URLs, Google Chrome/Edge on Web fails to seek accurately due to inaccurate byte-to-time estimation (since Chrome cannot do HTTP Range requests on blob URLs).

To resolve this:
- We use a hybrid playback system in `WebAudioEngine`.
- **WAV files**: Played using the **Web Audio API (`AudioBufferSourceNode`)** for sample-accurate seeking and zero playback drift.
- **MP3 and other files**: Played using the HTML5 `<audio>` element to preserve background playback and iOS compatibility.

**CRITICAL RULE**: Do not replace Web Audio API playback for WAV files with standard HTML5 `<audio>` element seeking, as it will re-introduce the audio-text desync bug.
